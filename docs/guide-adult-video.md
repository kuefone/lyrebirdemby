# 3. 🔞 成人影像 (Adult Video) {#av}

**【开篇】** {#av-opening}

欢迎来到数字收藏的"特殊领域"，一个充满激情与混乱的硬核世界。

成人影像，尤其是以**日本 AV (`JAV`)** 为代表的作品，呈现出一种既**高度标准化 (番号体系)** 又**极其混乱 (社区标签泛滥)** 的状态。而**欧美及其他地区的成人影像 (`Western Porn`)** 则遵循一套截然不同、**缺乏统一标识符**的命名逻辑，给自动化管理带来了巨大的挑战。

本章将**直接面向进阶收藏者**，深入剖析**各类成人影像**的命名逻辑。我们将**重点深挖 `JAV` 独有的“番号 (`ID Code`)”体系**（因为它是自动化整理的唯一钥匙），**同时详细拆解欧美成人内容的命名特点与整理难点**，并探讨相关的技术规格和社区文化。

最终目标是助您掌握如何利用这些规则，配合 `Stash` 等专业工具，来驯服这头“数据猛兽”，构建一个真正有序、可控、令人赏心悦目的“私人珍藏馆”。

::: warning 重要声明
1.  本章**不包含**香港"三级片 (`Category III Films`)"(其本质是电影, 应遵循 [电影篇](#movies-tv) 规则) 和日本"里番 (`Hentai`)"(其本质是动画, 应遵循 [动漫篇](#anime) 规则)。
2.  请确保您在收藏、浏览和分享相关内容时, **严格遵守当地的法律法规**。本指南仅作技术探讨和信息整理之用, 不鼓励或认可任何非法行为。
:::

### 3.1 🧠 核心逻辑：两大流派的根本差异 {#av-philosophy}

在开始整理之前，你必须明白，`JAV` 和 `Western` 是两种完全不同的“物种”，必须用两种截然不同的思路来对待。

::: details [+] 3.1 点击展开：番号中心制 (JAV) vs 场景中心制 (Western)
#### 3.1.1 🇯🇵 JAV：番号中心制 (ID-Centric) {#av-philosophy-jav}

* **核心**: **`番号 (ID Code)`**。
* **理念**: JAV 产业是**围绕“作品”**构建的。每部发行的作品（无论实体碟还是数字版）都会被赋予一个**近乎唯一**的身份证号（番号），例如 `SSIS-123`。
* **逻辑**: **抓住了番号，就抓住了全部**。文件名可以乱七八糟，演员可以不认识，但只要番号对了，就能通过数据库 (`JavBus`, `JavDB` 等) 查到这部作品的“户口本”——演员、厂商、系列、发行日期、标签、封面... 一切信息都挂在番号上。
* **整理难度**: **相对简单**。你的核心任务就是**从混乱的文件名中正确提取出番号**。

#### 3.1.2 🇺🇸 欧美 (Western)：场景中心制 (Scene-Centric) {#av-philosophy-western}

* **核心**: **`工作室 (Studio)` + `日期 (Date)` + `演员 (Performer)`**。
* **理念**: 欧美产业是**围绕“网站/场景”**构建的。内容以“**`Scene`**”（通常 10-30 分钟的片段）为单位，在各大付费网站 (Paysites) 上**按天或按周**发布。**它没有“番号”这种东西！**
* **逻辑**: **识别一个欧美 `Scene` 靠的是多重信息的组合**。`[工作室].[日期].[演员A, 演员B].[场景标题]` 是唯一能标识它的线索。
* **整理难度**: **地狱级**。因为缺乏统一 ID，文件名中任何一个信息的缺失、拼写错误（尤其是演员名）、日期格式混乱，都会导致自动化工具“抓瞎”，匹配失败或匹配错误。

::: danger 核心结论
* 整理 JAV，你的工具（如 `Stash`) 在做的是：`"文件名 -> 提取番号 -> 数据库查询番号 -> 找到唯一作品"`。
* 整理欧美片，你的工具在做的是：`"文件名 -> 提取工作室/日期/演员 -> 数据库模糊搜索 -> 返回 10 个可能的结果 -> 你懵了"`。
* **因此，这两类收藏必须使用不同的工具和策略来管理。**
:::
:::

### 3.2 🛠️ 社区生态：数据库、刮削器与命名乱象 {#av-community}

正因为 AV 收藏如此混乱，才催生了强大的社区工具来试图“拯救世界”。

::: details [+] 3.2 点击展开：专业工具、数据库与社区命名
#### 3.2.1 🗃️ 核心数据库 (Databases) {#av-community-databases}

* **JAV 数据库**:
    * **`JavBus` (javbus.com)**: 圈内最常用的“公交车”，更新快，信息全，有磁力链接索引。
    * **`JavDB` (javdb.com)**: 另一个强大的数据库，信息同样丰富，部分同好可能更喜欢它的标签体系。
    * **`JavLibrary` (javlibrary.com)**: 老牌数据库，资料库深厚，尤其是一些老片。
    * **`XCITY` / `DMM` / `FANZA`**: 日本**官方**的销售/数据库平台，信息最权威，但访问和刮削可能需要日本 IP 且格式规整。
    * `AirAV`: 台湾地区的数据库，繁体中文信息较多。
* **欧美数据库**:
    * **`ThePornDB (TPDB)` (theporndb.net)**: **欧美收藏的救星**。它试图效仿 `TMDb`，为**每一个**欧美 `Scene` (场景) 分配一个**唯一 ID**。`Stash` 等工具主要依赖它来刮削欧美内容。
    * **`PornDB` (via StashDB)**: `Stash` 社区维护的一个数据库，与 `Stash` 深度集成，数据质量高。
    * `IAFD` (Internet Adult Film Database): 老牌数据库，信息详尽，但更偏向传统 DVD 电影时代，对现代 WEB `Scene` 的更新可能不及时。

#### 3.2.2 🤖 自动化管理工具 (Scrapers & Managers) {#av-community-tools}

* **`Stash` (stashapp.cc)**:
    * **定位**: **AV 收藏的终极答案**，开源、跨平台、功能最强。
    * **优点**:
        * **通吃 JAV/Western**: 内置和社区插件支持连接上述**所有**数据库 (`JavBus`, `JavDB`, `TPDB` 等)。
        * **智能识别**: 不仅能用正则表达式提取 JAV 番号，还能对欧美文件名进行**模糊匹配**，甚至支持**视频指纹 (PHash / OSHash)** 识别（刮削“三无”文件）。
        * **精细化管理**: 强大的演员、标签、工作室管理系统，支持别名、层级、关系网络。
        * **场景标记 (`Scene Marking`)**: 可以在视频时间轴上**标记**不同演员的出现、不同动作的开始结束，功能极其变态。
    * **缺点**: **配置相对复杂**，需要一定的学习成本（尤其是刮削器配置）。资源消耗（尤其在首次扫描和生成预览图/截图时）可能较高。

* **Plex/Jellyfin/Emby 插件**:
    * **定位**: 在你现有的媒体库里“打补丁”。
    * **代表**: `PhoenixAdult (PA)` (Plex/Jellyfin), `XBMCnfoJAVImporter` (Kodi/Jellyfin/Emby, 专注 JAV NFO) 等。
    * **缺点**:
        * **配置繁琐/易失效**: 依赖插件，可能随版本更新而失效。
        * **刮削能力受限**: 识别率和功能性远不如 `Stash`。
        * **管理能力为零**: Plex 本身不是为 AV 管理设计的，无法进行精细的标签和演员管理。

* **`JavBus Manager` / `JavDB Manager`**:
    * **定位**: 专精 JAV 的轻量级管理工具。
    * **优点**: 简单易用，专注 JAV，刮削 `JavBus`/`JavDB` 准确率高。
    * **缺点**: **无法管理欧美内容**，功能单一。

::: danger 核心建议
**如果你是认真的，并且有 JAV 和欧美混合收藏，请一步到位，上 `Stash`。** 它虽然初学门槛高，但回报是指数级的。Plex 插件方案只适合轻度 JAV 用户，最终你还是会换到 Stash 的，别走弯路。
:::

#### 3.2.3 社区命名的“陷阱”与应对 {#av-community-naming-traps}

原始文件名是自动化整理的**最大敌人**。

* **JAV 的陷阱**:
    * **番号错误/混淆**: `SSNI-123` 写成 `SNIS-123`，或者 `h_123abc456` 这种在线番号格式混乱。
    * **过多“噪音”**: `[Thz.la][1080p.HEVC.AAC] SSIS-123 [亚洲巨乳美少女] [中文字幕].mkv`。这一堆中文和括号**极易干扰**刮削器提取 `SSIS-123` 这个核心。
* **欧美的陷阱**:
    * **信息缺失**: 只有演员名和标题，没有工作室和日期。
    * **信息错误**: 演员名拼错（`Liya Silver` vs `Lya Silver`)，日期写错。
    * **标题随意**: 同一个 `Scene`，官网叫 `A`，发布组叫 `B`，另一个网站叫 `C`。

::: tip 应对策略 (自动化流程)
1.  **初始文件名尽量简洁**:
    * **JAV**: 最好是 `ABCD-123-C.mkv` 或 `[S1] ABCD-123-C.mkv`。**保证番号清晰无误是第一要务**。
    * **欧美**: 尽量保留 `Studio.YY.MM.DD.Actors.Title` 的结构，**去除**下载网站、广告等无关字符。
2.  **依赖专业工具刮削**: 使用 `Stash`！
3.  **刮削与修正**:
    * `Stash` 扫描后，JAV 基本能自动识别 95% 以上。
    * 欧美内容可能只有 50% 识别率，剩下的大量需要**手动修正 (`Manual Edit`)** 或 **手动搜索匹配 (`Identify`)**（在 `Stash` 里搜索 `TPDB` 数据库，调整关键词找到正确的 `Scene`）。**这是整理欧美内容不可避免的宿命，必须接受！**
4.  **最终重命名**: 刮削成功后，让 `Stash` 按照你设定的**统一模板**自动重命名文件和文件夹。例如：
    * **JAV**: `[Studio] ID (Actress) [Resolution][Censored/Uncensored][Subbed]`
    * **欧美**: `[Studio] YYYY.MM.DD (Actors) - Title [Resolution]`
5.  **享受成果**: 通过 `Stash` 的 Web 界面，按演员、标签、工作室、评分等维度随意浏览你的“私人图书馆”。

**一句话总结：别信文件名里的演员和标签！那都是社区乱加的，仅供参考！一切以 `JavBus`/`JavDB`/`TPDB` 数据库刮削下来的为准！**
:::
:::

### 3.3 🇯🇵 JAV 深度解析：番号 (ID Code) {#av-jav-id}

这是 JAV 收藏的**基石**。搞懂了番号，你就掌握了打开 JAV 图书馆的钥匙。

::: details [+] 3.3 点击展开：番号的构成、变体与演化 (圣经级详解)
#### 3.3.1 什么是番号？(ID Code / Hinban) {#av-jav-id-what-is}

* **`番号 (ばんごう, Bangō)`**: 日语原意就是“编号”或“号码”（比如电话番号、部屋番号）。
* **AV 语境**: 特指 JAV 作品封面上那个独一无二的**识别码**，是厂商用于管理和发行的**产品编号**。
* **位置**: 通常在封面的左下角或右下角（没在就使劲找找）。
* **核心价值**: 它将一部作品与**厂商**、**系列**、**演员**、**发行日期**等所有关键信息**唯一绑定**。它是刮削器和数据库的**唯一索引**。

#### 3.3.2 番号的构成 {#av-jav-id-structure}

绝大多数 JAV 番号都遵循一个基本公式：

**`字母代码 (厂商/系列)` - `数字序号 (发行号)`**

* **`字母代码 (Prefix)`**:
    * **含义**: 通常是 2-5 位大写字母，代表**制作厂商 (Studio)** 或其旗下的**厂牌/系列 (Label/Series)**。
    * **举例**: `SSIS` (S1 厂牌), `MIDE` (Moodyz 厂牌), `IPX` (IdeaPocket 厂牌)。
    * **意义**: 这是识别作品“血统”的关键。看到 `MEYD` 就知道是“溜池ゴロー”的人妻系列，看到 `ADN` 就知道是 Attackers 的硬核剧情片。
* **`数字序号 (Number)`**:
    * **含义**: 通常是 3-5 位数字，代表该厂牌/系列下的**发行顺序号**。
    * **举例**: `SSIS-123`, `MIDE-999`。
    * **补零 (`Zero-padding`)**: 大部分厂商会用 `0` 把序号补齐到固定长度，比如 `001`, `023`。
    * **意义**: 序号通常是递增的，可以大致判断作品的新旧（但注意, **不能跨系列比较**！`MIDE-001` 可能比 `MIDD-999` 还新）。

#### 3.3.3 番号变体与特殊标识 (大全) {#av-jav-id-variants}

这部分就是老鸟的“内功”了。学会看懂番号前后的“花活儿”，能让你在下片前就洞悉一切。

* **🔲 格式变体 (Format Variants)**
    * **`BD (Blu-ray)`**: 番号中加入 `BD` 字样，表示这是**蓝光版**。
        * **示例**: `CWP-79` (DVD版) vs `CWPBD-79` (蓝光版)。
        * **`MKBD-S49`**: MUGEN 社的蓝光版。
        * **`HITMA-xxx`**: TMA 社的蓝光番号。
    * **`3D`**: 番号中加入 `3D` 字样，表示这是 **3D 版**。
        * **示例**: `SMDV-18` (2D版) vs `SM3D2DBD-18` (3D+2D 收藏版)。
* **🏪 发行渠道变体 (Channel Variants)**
    * **`R (Rental)`**: 番号带 `R` (前缀、后缀或无连字符)，表示这是**租赁版 (Rental Version)**。
        * **示例**: `XV-xxx` (贩卖版) vs `SRXV-xxx` (租赁版)。
        * **`84ABCD-123`**: 早期租赁版常见格式，前面的数字是渠道码。
        * **💡 区别**: 租赁版是面向线下租赁店的，内容可能与**零售版 (`Sell Version`)** 略有不同（比如时长较短、无特典影像、封面不同）。
    * **`h_ (DMM)`**: 番号以 `h_` 开头，是 **DMM 平台** (日本最大的数字内容分发商之一, 现属 FANZA) 的**下载版**标识。
        * **示例**: `h_113ssis001` (其中 `113` 是 DMM 内部给 S1 的厂商代码, `ssis001` 才是核心番号)。
* **🔢 特殊序号 (Special Numbers)**
    * **`SP` (Special)**: 特别篇，内容可能与主系列不同。
    * **`DX` (Deluxe)**: 豪华版，通常时长更长或有额外内容。
    * **`BOX`**: 多张光盘的套装或超长合集。
    * **`BEST` / `4H` / `8H`**: 精选集或超长总集篇 (俗称“素股合集”)。
    * **`KAN` (感谢祭)**: 厂商为回馈粉丝制作的特别企划,通常有多位演员参与。
    * **`PREMIUM`**: 有时也用于表示特别版。
    * **`ABCD-123.5`**: 番外篇或补充内容。
* **✂️ 剪辑版 (Edited Versions)**
    * **多番号共存**: 有时两家厂商会发行内容相似但**番号不同**的作品，此时要小心。
    * **示例 (来自 `6park`)**: `Carib-020913-260` (加勒比无码版) 可能是 `CWP-79` (CATWALK 有码版) 的**剪辑版**，内容不一定 100% 相同。

#### 3.3.4 番号演化与换代 (ID Evolution) {#av-jav-id-evolution}

番号不是一成不变的，厂商会因为各种原因更换番号，了解这个能帮你理清厂商的“家谱”。

* **原因 1：数字满了 (999 溢出)**
    * 当一个番号系列（比如 `xxx-999`) 发行完毕后，厂商会开一个**新的字母番号**，序号从 `001` 重新开始。
    * **经典案例 (来自 `6park`)**:
        * **IdeaPocket**: `IPTD-xxx` (用满 999) → 换代为 **`IPZ-xxx`**。
        * **S1**: `ONED-xxx` (用满 999) → 换代为 **`SOE-xxx`**。
* **原因 2：系列企划 (Series Planning)**
    * 厂商会为某个**特定企划**或**风格**定制**专属番号**，以示区分。
    * **示例**:
        * **SOD**: `SDMT-xxx` 是其著名的**素人企划**专用。
        * **IdeaPocket**: `SUPD-xxx` 是其“**DIGITAL CHANNEL (数字频道)**”系列专用。
        * **S1**: `SPS-xxx` 曾是为**瑠川リナ (Rukawa Rina)** 开设的“**TrueLove**”清纯系专属番号。
* **原因 3：厂商风格混乱 (a.k.a 蚊香社)**
    * **`Prestige (蚊香社)`**: (详见 3.4) 这家厂牌的番号体系极其混乱，堪称“**系列控**”。他们**几乎每个新系列都换一套新番号**（`ABS`, `ABP`, `DOM`, `YRZ`...），而不是像 S1 那样一个番号用到底。
    * **好处**: 看到番号字母大概就能猜到是啥类型（比如 `INU-xxx` 是硬核犬系）。
    * **坏处**: 番号多如牛毛，记起来费劲。

::: tip 逸闻趣事：看番号识“人生”
番号不仅是编号，有时还能看出**演员的生涯轨迹**。
* **瑠川リナ (Rukawa Rina)**: 早期番号是 S1 的清纯系专属 `SPS-001` (开系掌门)；当她的番号变为 `SOE-663` 时，就意味着她已从“清纯派毕业”，加入 S1 的“常规军”行列，风格可能随之改变。
* **初音みのり (Hatsune Minori)**: 在 S1 时期是 `ONED-864` (S1 育成偶像，小嫩妹)；后来移籍到 IdeaPocket，番号变为 `IPZ-074`，风格也变成了“受辱他人之妻”。

**看懂番号的演变，你就能像个老司机一样，从番号变化中品读出演员的“进化”或“堕落”（bushi），这才是收藏的醍醐味啊！**
:::

### 3.4 🏛️ JAV 厂商/集团/厂牌流派 (究极大全) {#av-jav-studios}

这是本章的**核心**，也是最庞杂的部分。了解厂商的“秉性”和“马甲”（厂牌），是根据口味精准“寻宝”或“排雷”的关键。

::: details [+] 3.4 点击展开：JAV 厂商流派究极圣经
#### 3.4.1 北都集团 (Hokuto Corporation / CA / 圈内俗称：骑马) {#av-jav-studios-hokuto}

**地位**: 日本 AV 界的"巨无霸"，市场占有率遥遥领先。前身是 `Outvision` / `Will Group`。旗下厂牌（马甲）多如牛毛，覆盖了几乎所有题材和风格，从顶级制作到小众企划，应有尽有。

* **`S1 No.1 Style (エスワン)`**
    * **特色与风格**: CA 集团的**王牌旗舰**，也是整个行业的标杆。以**超高画质、精良制作、豪华布景**和**顶级专属女优**著称。作品风格"正统"，画面明亮，剧情企划感强。新人女优若能在 S1 出道, 通常被视为最高起点。马赛克类型通常为厚码。
    * **知名番号系列**:
        * `SSIS`: **现役旗舰番号**，顶级专属的主打作。
        * `SSNI`: 并行的旗舰番号。
        * `SNIS`: 稍早期的旗舰番号。

::: tip “骑马” (Kiba) 是什么意思？
在深入厂商之前，你可能会在你给的 `6park` 资料（[file: image_c694cb.png]）和很多社区里看到厂商后面跟着 `(骑马)` 这两个字。这是圈内的一个“黑话”！

* **`骑马 (きば, Kiba)`**: 指的是**有马赛克 (`Censored`)** 的作品，即“**骑兵**”。
* **`步兵 (ほへい, Hohei)`**: 相对的，指的是**无马赛克 (`Uncensored`)** 的作品。

这个叫法源于早期社区的调侃（具体来源已不可考，一说是“有马”听起来像“有码”的谐音，一说是指骑兵作战时“看不清脸”或“穿着盔甲”...总之就是个梗）。

所以，看到 `S1 (骑马)`，意思就是 **“S1 是一个有码厂商”**。咱们接下来要讲的大部分 JAV 正规厂商，默认都是“骑马”。后面会单独开一章讲“步兵”厂商。
:::

### 3.6 🇺🇸 欧美及其他地区 (Western Porn etc.) {#av-western}

::: details [+] 3.6 点击展开：欧美内容的“场景中心制”与管理噩梦
#### 3.6.1 欧美内容命名特点 (Scene-Centric) {#av-western-naming}

如“核心逻辑”中所述, 欧美成人影像 **缺乏类似 JAV 番号的统一 ID 体系**。其命名和识别主要依赖以下元素的组合, 通常以"**场景 (`Scene`)**"为单位发布和管理。

* **`工作室/网站 (Studio / Site)`**:
    * **重要性**: **极其重要**，是分类和识别的核心依据之一。
    * **常见示例**:
        * **大型集团**: `Brazzers`, `Reality Kings`, `Naughty America`, `BangBros`, `Pornhub Premium` (旗下众多子品牌如 `TeamSkeet`, `MILFY`, `Fake Taxi`...)
        * **高端/唯美**: `Blacked`, `Vixen`, `Tushy`, `Deeper`, `Slayed`... (Greg Lansky 的品牌宇宙)
        * **硬核/导演中心**: `Jules Jordan Video`, `Evil Angel`, `Manuel Ferrara`...
    * **文件名体现**: 通常放在文件名的**最前面**，使用标准名称或公认缩写 (如 `RK`)。

* **`发布日期 (Date)`**:
    * **重要性**: **非常重要**。是区分同一工作室、同一演员不同场景的关键时间戳。
    * **格式**: 最常用 `YY.MM.DD` (如 `23.10.25`) 或 `YYYY-MM-DD` (`2023-10-25`)。**保持格式统一**有助于刮削。

* **`演员名 (Performers)`**:
    * **重要性**: **核心信息之一**。观众通常会根据喜爱的演员来选择作品。
    * **痛点**: **名字的准确性和完整性是社区命名的最大痛点**。可能只写女演员；可能使用昵称/艺名；可能拼写错误；演员可能更改艺名... 这些都会导致刮削困难。

* **`标题/场景描述 (Title / Scene Description)`**:
    * **重要性**: 描述该视频片段的主要内容或主题。
    * **特点**: 通常比较**长**，**描述性强**，但**极其不规范和随意**。对于刮削来说, 标题的**权重通常低于**工作室+日期+演员。

* **技术规格 (`Technical Specs`)**: `1080p`, `4K`, `HEVC`, `WEB` 等, 同电影篇。

#### 3.6.2 常见欧美命名结构 {#av-western-naming-structure}

**`Studio.YY.MM.DD.Actress1.[Actress2.And.Actor].Scene.Title.[Specs]-[Group].ext`**

* **注意**: 并非所有部分都存在, 顺序也可能变化很大。**核心是 `Studio` + `Date` + `Performers`**。
* **示例**:
    * `Blacked.23.10.20.Gianna.Dior.And.Jax.Slayher.Forbidden.Fruit.XXX.1080p.WEB.h264-GRP.mp4`
    * `TeamSkeet.23.11.05.Kylie.Rocket.Daddy.Likes.It.Rough.XXX.4K.HEVC.mp4`
    * `Vixen.23.09.15.Liya.Silver,Blake.Blossom.Office.Temptations.XXX.2160p.WEB.HEVC.mp4`

#### 3.6.3 欧美刮削与管理难点 {#av-western-scraping}

::: danger 警告：欧美整理 = 肝帝模式
由于缺乏统一 ID, 自动化刮削和管理欧美成人影像比 JAV 面临更多挑战, 需要更强大的工具和**大量的人工干预**。

1.  **识别困难**: 刮削器无法像 JAV 那样通过单一、精确的番号来匹配数据库。它需要**综合分析**文件名中的**多个字段**(工作室、日期、演员、标题), 并与数据库中的场景信息进行**模糊匹配 (`Fuzzy Matching`)**。文件名中任何一个关键信息的错误或缺失都可能导致匹配失败。
2.  **数据库依赖**: 刮削的准确性高度依赖于 **`ThePornDB (TPDB)`**。但其数据也依赖社区贡献, 可能存在错误、遗漏或延迟。
3.  **演员别名 (`Aliases`)**: 演员可能使用不同艺名, 或者文件名中的拼写与数据库不一致, 或者演员后来改名, 这些都会导致匹配困难。
:::

### 3.7 📺 通用技术规格与总结 {#av-tech-summary}

最后，我们来总结一下 AV 收藏中常见的技术规格和特殊标签。

::: details [+] 3.7 点击展开：AV 的技术规格、字幕与结语
#### 3.7.1 技术规格特殊性 (Technical Specs in AV) {#av-tech-specs}

* **分辨率与画质 (`Resolution & Quality`)**:
    * **`1080p`**: **JAV 和欧美的绝对主流**。
    * **`4K / 2160p`**:
        * **欧美**: **非常普遍且真实性高**。很多工作室 (如 `Blacked`, `Vixen`) 直接使用 4K 甚至更高规格拍摄, 原生 4K 内容丰富。
        * **JAV**: **逐渐增多, 但需警惕"伪4K"**。以 `FALENO` 为代表的新厂商主打原生 4K, 但很多 4K 资源是 **`1080p` 拉伸 (`Upscale`)** 的, 体积巨大但细节提升有限。
    * **`720p`**: 常见于在线平台 (`Heyzo`, `FC2`) 或老旧资源。
* **色深与动态范围 (`Bit Depth & HDR`)**:
    * **`8bit SDR`**: **绝对主流**。99.9% 的 AV 内容都是 `8bit SDR`。
    * **`10bit / HDR`**: **极其罕见**。这行当主要看“画面内容”，技术规格通常够用就行，成本和受众决定了 HDR 并非刚需。
* **视频编码 (`Codec`)**:
    * **`H.264 (AVC)`**: 仍然**非常常见**，尤其是 1080p 的 WEB 源。
    * **`H.265 (HEVC)`**: **节省空间的首选**。在 4K 内容中几乎是标配，高质量的 1080p `BDRip` (JAV) 或 `WEBRip` (欧美) 也会采用。
* **音频 (`Audio`)**:
    * **声道**: 99.9% 都是 **`2.0` (立体声)**。
    * **格式**: 99.9% 都是**有损**格式, 如 `AAC` (最常见) 或 `MP3` (老资源)。别指望 `FLAC` 或 `Atmos` 了。
* **VR 格式 (`Virtual Reality`)**:
    * **分辨率**: 通常很高, 对应**展开后**的全景画面, 如 `3840x1920`, `5400x2700`, `7680x3840` (`8K`)。
    * **视角**: **`180`** (度) 或 `360` (度)。`180度` **前向视角**更常见, 符合人眼观看习惯, 也能节省一半的数据量。
    * **布局 (`Layout`)**: **`SBS`** (Side-by-Side, 左右格式) 或 `TB` / `OU` (Top-and-Bottom / Over-Under, 上下格式)。`SBS` 最常见。
    * **文件名**: 常带有 `VR`, `180`, `SBS`, `8K` 等标识。
    * **播放**: 需要 VR 头显和 `DeoVR` 等专业播放器, 并正确设置投影方式。

#### 3.7.2 字幕与审查标签 (`Subtitles & Censorship`) {#av-tech-subs}

* **审查状态 (Censorship)**:
    * (无标签) 或 `-C` (紧跟番号): **有码 (`Censored`)**。JAV 的默认和合法状态, 关键部位有马赛克。
    * `-U`: **无码 (`Uncensored`)**。来源通常是海外版、泄露版或特定无码平台 (`1Pondo`, `Caribbeancom` 等)。
    * `-F` / `Leaked`: **泄露版**。特指厂商内部母带流出, 画质通常较差。
    * `Decensored`: **AI去码/破解码**。第三方技术处理版, 并非真无码, 效果不定。

* **字幕 (`Subtitles`)**:
    * **`_C` / `-C` (在文件名末尾)**: **中文字幕**。JAV 最常见的字幕标识, 通常是**内嵌 (Hardsub)**。
    * `-CHS` / `.Chs`: **简体中文**字幕。
    * `-CHT` / `.Cht`: **繁体中文**字幕。
    * **欧美**: 带字幕的极少。

::: warning 命名冲突：-C
**注意！`-C` 这个标签是最容易混淆的！**
* 紧跟在番号后面, 如 `SSIS-123-C.mkv`, 它 **99%** 的意思是 **`Censored` (有码)**。
* 出现在文件名**末尾**, 如 `SSIS-123.1080p.mkv._C` 或 `SSIS-123-C.chs.mkv` (C 和 chs 一起出现), 它 **99%** 的意思是 **`Chinese Subbed` (中文字幕)**。

**`Stash` 等工具的正则表达式通常能智能区分这两种情况。**
:::
